# Пояснение по реализации экспорта, импорта и размеров

## Общая идея

В программе есть три связанные подсистемы:

1. экспорт фигур в DXF;
2. импорт фигур из DXF;
3. внутренняя система размеров.

Они связаны между собой через общую модель фигур: все примитивы хранятся в `ShapeManager`, рисуются единым рендерером и имеют одинаковую базовую схему работы через методы `draw()`, `get_bounds()`, `distance_to_point()`, `to_dict()` и `from_dict()`.

Важно: размеры в приложении реализованы как полноценные внутренние объекты, но в DXF сейчас экспортируются и импортируются только геометрические примитивы. То есть размеры работают внутри программы, но не сохраняются в DXF как сущности `DIMENSION`.

---

## 1. Экспорт в DXF

### Где реализован

Экспорт находится в файле `exporters/dxf_exporter.py`, основной класс — `DxfExporter`.

Из главного окна экспорт вызывается методом `GeometryApp._on_export_dxf()`, а команда "Сохранить" сейчас просто вызывает этот же DXF-экспорт.

### Общая схема работы

Метод `export()` получает:

- путь к файлу;
- список фигур;
- список стилей линий;
- отступ `margin`.

Далее он:

1. вычисляет границы всех фигур через `get_bounds()`;
2. расширяет их на заданный отступ;
3. открывает файл в кодировке `cp1251`;
4. вручную записывает стандартные секции DXF:
   - `HEADER`
   - `CLASSES`
   - `TABLES`
   - `BLOCKS`
   - `ENTITIES`
   - `OBJECTS`
   - `EOF`

Таким образом экспорт у вас сделан не через сторонний writer, а через прямую генерацию DXF-текста.

### Что записывается в HEADER

В секции `HEADER` задаются:

- версия DXF;
- кодовая страница;
- единицы вставки;
- служебный `HANDSEED`;
- пределы чертежа;
- масштаб типов линий.

Это нужно, чтобы файл корректно открывался в CAD-системах и имел правильные границы модели.

### Таблицы DXF

В секции `TABLES` создаются стандартные таблицы:

- `VPORT`
- `LTYPE`
- `LAYER`
- `STYLE`
- `VIEW`
- `UCS`
- `APPID`
- `DIMSTYLE`
- `BLOCK_RECORD`

Наиболее важные для вашей программы:

#### `LTYPE`

Внутренние стили линий преобразуются в DXF-типы:

- `solid -> Continuous`
- `dashed -> HIDDEN`
- `dashdot -> CENTER`
- `dashdotdot -> PHANTOM`
- `wavy -> WAVES`
- `broken -> ZIGZAG`

Для штриховых типов дополнительно формируется паттерн штрихов и пробелов.

#### `LAYER`

Создаются:

- обязательный слой `0`;
- слой `Defpoints`;
- слои на основе известных стилей линий.

Слой `Defpoints` используется для служебных непечатаемых точек, которые помогают CAD-системе корректно определить границы модели.

#### `DIMSTYLE`

В таблицу добавляется стандартный `DIMSTYLE`, но это пока не означает поддержку полноценного DXF-размера. Таблица нужна скорее для корректности структуры файла.

### Как экспортируются фигуры

Фигуры записываются в секции `ENTITIES`.  
Для этого у экспортера есть общий диспетчер `_write_entity()`, который по типу фигуры вызывает нужный метод:

- `POINT`
- `LINE`
- `CIRCLE`
- `ARC`
- `ELLIPSE`
- `LWPOLYLINE`
- `SPLINE`

Поддерживаются следующие внутренние примитивы:

- `Point`
- `Segment`
- `Circle`
- `Arc`
- `Ellipse`
- `Rectangle`
- `Polygon`
- `Spline`

### Общие свойства сущности

Перед записью геометрии вызывается `_write_common_entity_props()`, который задаёт:

- тип DXF-сущности;
- новый `handle`;
- принадлежность к `Model Space`;
- слой;
- цвет;
- тип линии;
- масштаб типа линии.

Слой теперь берётся так:

1. если у фигуры есть `layer`, используется он;
2. иначе используется `line_style_name`.

Это важно для размеров, потому что у них есть собственный слой `"Размеры"`.

### Как задаются границы файла

После записи всех сущностей экспортер специально добавляет две `POINT` на слой `Defpoints` в крайних точках области `ext_min` и `ext_max`.  
Это сделано для тех CAD-систем, которые ориентируются не только на заголовок DXF, но и на реальную геометрию в модели.

### Ограничение экспорта

Сейчас в DXF не экспортируются:

- `LinearDimension`
- `RadialDimension`
- `AngularDimension`

Причина в том, что `_write_entity()` пока не содержит веток для этих типов. То есть экспорт касается только геометрических объектов.

## 2. Импорт из DXF

### Где реализован

Импорт находится в файле `importers/dxf_importer.py`, основной класс — `DxfImporter`.

В приложении он запускается из `GeometryApp._on_open()`.

### Общая схема работы

Импорт использует библиотеку `ezdxf`.

Метод `import_file()`:

1. открывает DXF через `ezdxf.readfile()`;
2. получает `modelspace()`;
3. разворачивает все блоки `INSERT` через `explode()`;
4. проходит по всем сущностям;
5. для каждой сущности определяет:
   - слой;
   - стиль линии;
   - цвет;
6. создаёт внутренние объекты программы;
7. возвращает список фигур.

### Восстановление слоя, стиля и цвета

Это делается в `_get_entity_style()`.

#### Слой

Слой читается из `entity.dxf.layer`.  
Если в тексте слоя встречаются AutoCAD-последовательности вида `\U+XXXX`, они декодируются в обычные символы.

#### Стиль линии

Из DXF-типа линии восстанавливается внутреннее имя стиля.  
Если у сущности стоит `BYLAYER`, тип линии берётся из слоя.

Дополнительно есть эвристика:

- для `Continuous` учитывается толщина линии;
- по толщине определяется, это основная или тонкая линия.

#### Цвет

Цвет восстанавливается в порядке приоритета:

1. `true_color`;
2. цвет слоя, если сущность работает в режиме `ByLayer`;
3. обычный индексный ACI-цвет.

Внутри программы цвет хранится как строка `#RRGGBB`.

### Какие DXF-сущности поддерживаются

Импортируются:

- `LINE`
- `POINT`
- `CIRCLE`
- `ARC`
- `ELLIPSE`
- `LWPOLYLINE`
- `POLYLINE`
- `SPLINE`

### Как создаются внутренние объекты

Преобразование выполняется в `_create_shape_from_entity()`.

Логика такая:

- `LINE` -> `Segment`
- `POINT` -> `Point`
- `CIRCLE` -> `Circle`
- `ARC` -> `Arc`
- `ELLIPSE` -> `Ellipse`

Для полилиний и сплайнов логика сложнее:

- если объект процедурный и содержит много точек, он может быть упрощён;
- если полилиния слишком тяжёлая, применяется децимация;
- если полилиния замкнута, она может раскладываться в набор отрезков;
- если доступны виртуальные примитивы, из них извлекаются `LINE` и `ARC`.

Такой подход уменьшает нагрузку на рендер и помогает аккуратно восстанавливать геометрию из разных DXF-источников.

### Дополнительная обработка после импорта

После чтения фигур в `main.py` выполняется восстановление реальных имён стилей:

- строится карта очищенных имён;
- если имя слоя или стиля было упрощено при экспорте, оно возвращается к исходному виду;
- если точного совпадения нет, используется простая эвристика с заменой `_` на пробелы.

### Ограничение импорта

Сейчас из DXF не импортируются размерные сущности `DIMENSION`.  
Если в файле есть размеры, программа пока не преобразует их во внутренние `LinearDimension`, `RadialDimension` или `AngularDimension`.

## 3. Система размеров

### Общая структура

Размеры реализованы в файле `shapes/dimension.py`.

Иерархия классов:

- `Dimension` — базовый класс;
- `LinearDimension` — линейный размер;
- `RadialDimension` — радиус/диаметр;
- `AngularDimension` — угловой размер.

Все они наследуются от общего `Shape`, поэтому ведут себя как обычные фигуры:

- рисуются на холсте;
- выделяются;
- имеют границы;
- умеют сериализоваться;
- могут иметь контрольные точки.

### Базовый класс `Dimension`

В базовом классе собраны общие параметры оформления:

- цвет;
- стиль линии;
- переопределение текста;
- префикс;
- размер шрифта;
- тип стрелки;
- размер стрелки;
- параметры полки;
- слой `"Размеры"`.

Также здесь реализованы:

- отрисовка стрелок;
- формирование текста размера;
- получение параметров линии из стиля;
- общая сериализация через `_base_dict()` и `_apply_base_dict()`.

### Линейный размер

`LinearDimension` хранит две точки измерения и `offset`.

Поддерживаются три режима:

- выровненный;
- горизонтальный;
- вертикальный.

Размер умеет:

- автоматически вычислять измеряемую величину;
- строить размерную линию в зависимости от режима;
- держать ассоциативную связь с базовыми фигурами через `base_shape_id` и индексы snap-точек;
- обновлять свои точки при изменении исходной геометрии;
- редактироваться через контрольные точки текста и опорных точек.

### Радиальный размер

`RadialDimension` используется и для радиуса, и для диаметра.  
Это определяется флагом `is_diameter`.

Он хранит:

- центр;
- радиус;
- угол размерной линии;
- режим отображения.

Поддерживаются режимы:

- `leader` — полка;
- `aligned` — по линии;
- `outside` — наружный диаметр.

Если размер связан с окружностью или дугой, он обновляет центр и радиус по базовой фигуре, то есть работает ассоциативно.

### Угловой размер

`AngularDimension` хранит:

- вершину угла;
- две точки лучей;
- радиус размерной дуги;
- флаг выбора малого или большого угла;
- ссылки на базовые отрезки.

При создании углового размера:

1. пользователь выбирает два отрезка;
2. ищется их общая вершина или пересечение продолжений;
3. определяется, какие концы отрезков задают нужные лучи;
4. создаётся размер.

Если базовые отрезки меняются, размер пересчитывает вершину и направления лучей.

## 4. Создание и редактирование размеров

### Инструмент размеров

Основная логика создания находится в `tools/dimension_tool.py`, класс `DimensionTool`.

Поддерживаемые режимы:

- `linear`
- `radius`
- `diameter`
- `angular`

Инструмент использует объектные привязки через `SnapManager`, поэтому размеры создаются по точным snap-точкам.

### Параметры по умолчанию

Инструмент хранит набор параметров, которые применяются к новым размерам:

- режим линейного размера;
- оформление радиального;
- шрифт;
- тип стрелки;
- размер стрелки;
- префикс;
- параметры выноса.

Эти настройки задаются в левой панели интерфейса и применяются к новому размеру через `_apply_defaults()`.

### Редактирование

После создания размер можно редактировать как обычную фигуру:

- перетаскивать целиком;
- редактировать контрольные точки;
- менять свойства через правую панель.

В `SelectTool` и `EditTool` размеры используют общий механизм:

- `get_control_points()`
- `move_control_point()`

Это обеспечивает единый подход к редактированию геометрии и размеров.

## 5. Сериализация размеров внутри приложения

Хотя кнопки "Открыть" и "Сохранить" сейчас работают через DXF, в проекте уже есть внутренняя сериализация всех фигур:

- `ShapeManager.export_to_list()`
- `ShapeManager.import_from_list()`

Для размеров реализованы собственные:

- `to_dict()`
- `from_dict()`

для всех трёх типов:

- `LinearDimension`
- `RadialDimension`
- `AngularDimension`

Это значит, что программа уже готова к добавлению отдельного внутреннего формата проекта, в котором размеры можно будет сохранять полностью, вместе с оформлением и ассоциативными связями.

---

## 6. Итог

Сейчас в программе хорошо реализованы:

- DXF-экспорт геометрических примитивов;
- DXF-импорт геометрических примитивов;
- преобразование слоёв, цветов и стилей линий;
- собственная внутренняя система размеров;
- ассоциативность размеров внутри приложения;
- редактирование размеров через контрольные точки и свойства.

Основное текущее ограничение:

- размеры работают внутри программы, но не экспортируются и не импортируются как полноценные DXF-размеры.

Если развивать систему дальше, есть два естественных пути:

1. добавить собственный формат проекта для полного сохранения всех объектов, включая размеры;
2. отдельно реализовать поддержку DXF-сущностей `DIMENSION`.
